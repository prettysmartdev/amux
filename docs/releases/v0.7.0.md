# Release v0.7.0

## Features

- **Multi-agent workflows**: Each workflow step can now specify a different agent via an `Agent:` field in the step header.
  - ```markdown
    ## Step: plan
    Prompt: Produce an implementation plan.

    ## Step: implement
    Depends-on: plan
    Agent: codex
    Prompt: Implement the plan.

    ## Step: review
    Depends-on: implement
    Agent: claude
    Prompt: Review the implementation for correctness and style.
    ```
  - `Agent:` is optional. Steps without it use the workflow default agent — repo config, then global config, then the `--agent` flag if passed.
  - The `--agent` flag on `implement --workflow` sets the default for steps that do not specify an agent. It does **not** override steps with an explicit `Agent:` directive.
  - `Agent:` must appear after `Depends-on:` and before `Prompt:`; an `Agent:` appearing after `Prompt:` is treated as prompt text.
  - Unknown agent names in `Agent:` fields are caught at parse time, before any containers run.

- **Workflow agent pre-flight check**: Before the first step runs, amux validates that every agent image required by the workflow exists. Missing images are handled interactively:
  1. amux offers to download the agent Dockerfile template and build the image.
  2. If the user declines, amux offers to substitute the default agent for those steps instead.
  3. If both are declined, the workflow does not start.
  - All agents are checked before execution begins — no mid-workflow surprises.
  - If an agent's Dockerfile download or image build fails, amux surfaces the error and does not start the workflow. Partial Dockerfiles are removed.

- **`--agent` flag extended to workflow pre-flight**: When `implement --workflow --agent <name>` is used, `<name>` becomes the effective default for steps without an explicit `Agent:` field and is included in the pre-flight check.

- **"Same container" disabled across agent boundaries**: In the workflow control board, the **↓** ("next step: same container") action is disabled when the next step requires a different agent than the current step. The TUI renders it greyed out with a message:
  ```
  Next step uses agent 'codex'; cannot reuse current 'claude' container.
  ```
  In command mode the prompt is skipped and the explanation is printed instead. The **→** ("new container") action always works regardless of agent.

- **`amux ready --build` with multiple agent Dockerfiles**: When `.amux/` contains Dockerfiles for more than one agent, `ready --build` now prompts whether to build all agent images or only the configured default:
  ```
  Found 2 agent Dockerfiles:
    claude  (default)
    codex   (extra)

  Build all agent images, or only the default (claude)? [all/default]:
  ```
  Previously, `--build` silently rebuilt every agent image found without asking.

- **`--no-cache` applies to all builds in `ready`**: The `--no-cache` flag now extends to every `docker build` invocation during `amux ready`, including the project base image and all agent images. Previously it was not consistently applied to all images in the build sequence.

## Improvements

- **Prominent build markers**: Each image build started by `amux ready` (or `amux init`) is framed with clear start and end markers in the output, making it easy to follow multi-image build sequences:
  ```
  ══════════════════════════════════════════════════
    Building agent image: amux-myproject-codex:latest
  ══════════════════════════════════════════════════
  ```
  This applies to all build paths: `--build`, `--refresh`, `--no-cache`, and initial `init` builds.

- `WorkflowStepState` gains an `agent: Option<String>` field (serialized to JSON). The field uses `#[serde(default)]` so existing state files without the field deserialize without error. When resuming a workflow, the persisted per-step agent assignment takes precedence; a warning is printed if the CLI `--agent` flag differs from the persisted default.

- `src/commands/agent.rs` gains `ensure_agent_available(git_root, agent_name, out, ask_fn) -> Result<bool>`. The function checks for the agent Dockerfile, prompts via the supplied `ask_fn`, downloads and builds if accepted, and returns `true` (ready) or `false` (user declined). Used by both the `chat`/`implement` single-step path and the workflow pre-flight check.

- Per-agent Dockerfile download URLs are defined as `AGENT_DOCKERFILE_URLS: &[(&str, &str)]` in `src/commands/agent.rs`, co-located with `resolve_agent_image_and_dockerfile()`.

- `WorkflowStep` and `WorkflowStepState` in `src/workflow/parser.rs` and `src/workflow/mod.rs` both gain `agent: Option<String>`. `WorkflowState::new()` copies the parsed agent through to state; `validate_resume_compatibility()` does not check agent (it is a runtime hint, not a structural dependency).
