# Release v0.6.0

## Features

- **Modular Dockerfiles**: The container image setup is now split into two layers, separating project tooling from agent tooling.
  - **`Dockerfile.dev`** (at the Git root) is the _project base image_ — OS, language runtimes, build tools, and test dependencies specific to your project. Produces `amux-{project}:latest`.
  - **`.amux/Dockerfile.{agent}`** is the _agent image_ — extends the project base with AI agent tooling (Claude Code, Codex, OpenCode, Maki, or Gemini) and creates the non-root `amux` user. Produces `amux-{project}-{agent}:latest`. This is the image used by `chat` and `implement`.
  - `amux init` writes both files and builds both images in sequence.
  - `amux ready` verifies and builds both images. `--build` rebuilds both the project base and every agent dockerfile found in `.amux/`.
  - `--refresh` runs the audit agent against `Dockerfile.dev` only (project dependencies); agent dockerfiles are managed by templates and not modified by the audit.
  - Both files should be committed to source control — teammates share the same setup on clone.

- **`--agent <name>` flag on `chat` and `implement`**: Override the configured agent for a single session without modifying any config file.
  - ```sh
    amux chat --agent codex
    amux implement 0050 --agent gemini
    ```
  - If the requested agent image does not exist, amux offers to download the template and build both images (project base first, then agent image) before launching.
  - Unknown agent names exit immediately with a list of valid options: `claude`, `codex`, `opencode`, `maki`, `gemini`.

- **Migration path for existing single-file repos**: Repos with a `Dockerfile.dev` that bundles agent tooling (the pre-0.6 layout) are automatically detected. `amux ready` offers a guided migration:
  1. Recreates `Dockerfile.dev` with a minimal project base template
  2. Writes `.amux/Dockerfile.{agent}` from the agent template
  3. Builds both images
  4. Runs the audit agent to restore project dependencies in `Dockerfile.dev`

  If migration is declined, the existing image continues to work with a deprecation warning. `amux chat` and `amux implement` direct users to run `amux ready` to migrate, or pass `--no-migrate` to proceed with the legacy image for that session.

## Improvements

- `agent_image_tag()` added as a free function in `src/runtime/mod.rs` alongside `project_image_tag()`, producing `amux-{project}-{agent}:latest` given a git root path and agent name.
- `write_dockerfile()` in `src/commands/init.rs` split into `write_project_dockerfile()` (writes `Dockerfile.dev` from the project base template) and `write_agent_dockerfile()` (writes `.amux/Dockerfile.{agent}`, substituting `{{AMUX_BASE_IMAGE}}` with the project base tag). Each function is idempotent — neither overwrites an existing file.
- Agent dockerfiles now use `{{AMUX_BASE_IMAGE}}` as their `FROM` directive instead of a hardcoded base, allowing any project base to be used without modifying the agent template.
- `src/commands/agent.rs` (`run_agent_with_sink()`) now resolves `agent_image_tag()` as the launch image and accepts an `agent_override: Option<String>` parameter threaded from `--agent` on `chat` and `implement`.
- `amux ready` build sequence updated to: (1) ensure `Dockerfile.dev` exists, (2) ensure `.amux/Dockerfile.{agent}` exists, (3) build project base image, (4) build agent image.
- A new `templates/Dockerfile.project` embedded template provides the project base starting point: `debian:bookworm-slim` with `git`, `curl`, `make`, and `ca-certificates`.
