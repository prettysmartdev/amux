# Implementation Plan: 0028 — LSP Recommendation Suppression

## Problem Statement

Claude Code shows LSP (Language Server Protocol) installation recommendation dialogs inside
agent containers. These are noisy because containers have no IDE and no pre-installed language
servers. amux is supposed to suppress them; it doesn't currently work.

---

## Root Cause Analysis

### What exists today

`src/docker/mod.rs` contains `disable_lsp_recommendations()`, called from both
`HostSettings::prepare()` and `HostSettings::prepare_to_dir()`. It writes:

```json
{ "lspRecommendationDisabled": true }
```

into the temp `dot-claude/settings.json` that gets bind-mounted as `/root/.claude/settings.json`
inside the container.

### Root Cause 1 — Wrong settings key (primary)

`lspRecommendationDisabled` is **not a documented Claude Code setting**. Research against the
Claude Code npm package, GitHub changelog, and issue tracker found zero references to this key.
It has no effect. The correct key (or mechanism) is unknown from documentation alone and must
be determined empirically (see Investigation below).

### Root Cause 2 — No settings applied when host has no `~/.claude.json`

`HostSettings::prepare()` returns `None` when `~/.claude.json` is absent on the host. When
`host_settings` is `None`, `append_settings_mounts` is never called, so **no** `/root/.claude/`
or `/root/.claude/settings.json` is mounted into the container at all. LSP suppression is
therefore entirely skipped for users who have not yet logged in to Claude on the host machine.

### Root Cause 3 — No test coverage for `disable_lsp_recommendations`

The function has no dedicated unit test. There is nothing to catch a regression if the key
changes, or to verify the file-creation path when `settings.json` is absent.

---

## Investigation Step (must be done before coding)

Because the correct setting key is unknown, it must be determined empirically before writing
the fix. **One developer** should run this investigation and record the result in this plan.

### Method A — Observe what Claude writes after dismissing the dialog

1. Start a fresh container with no `~/.claude/settings.json` mounted (or an empty one):
   ```
   docker run --rm -it \
     -v $(pwd):/workspace \
     <amux-image> bash
   ```
2. Inside the container, run `claude` interactively.
3. Wait for or trigger the LSP recommendation dialog (open a `.py` or `.ts` file).
4. Dismiss the dialog ("Don't show again" or equivalent).
5. `cat ~/.claude/settings.json` and record the key(s) that were added.

### Method B — Search the minified Claude Code bundle

The `@anthropic-ai/claude-code` npm package ships a minified bundle. Search it for strings
near "lsp" and "recommend":

```
npm pack @anthropic-ai/claude-code
tar xf *.tgz
grep -o '"[a-zA-Z]*[Ll][Ss][Pp][a-zA-Z]*"' package/cli.js | sort -u
grep -o '"[a-zA-Z]*[Rr]ecommend[a-zA-Z]*"' package/cli.js | sort -u
```

### Likely candidates to test

Based on Claude Code naming conventions:
- `"hasShownLspRecommendation"` — one-time flag set after first showing
- `"lspRecommendationDismissed"` — explicit user dismissal
- `"suppressLspRecommendations"` — general suppression flag

### Fallback mechanism

If no settings.json key exists (Claude Code always checks at runtime), use the environment
variable approach. Research found that `ENABLE_LSP_TOOL=1` **enables** the LSP connection
feature; setting it to `0` or omitting it does not appear to suppress recommendation dialogs.
However, if empirical testing confirms that an env var can suppress recommendations, inject it
via `env_vars` in the container run args rather than via settings.json.

## Implementation

### Step 1 — Update `disable_lsp_recommendations` with the correct key

Once the key is confirmed from the investigation above, update `docker/mod.rs`:

```rust
fn disable_lsp_recommendations(claude_dir: &Path) -> std::io::Result<()> {
    let settings_path = claude_dir.join("settings.json");
    let mut settings: serde_json::Value = if settings_path.exists() {
        let raw = std::fs::read_to_string(&settings_path)?;
        serde_json::from_str(&raw).unwrap_or(serde_json::json!({}))
    } else {
        serde_json::json!({})
    };
    if let Some(obj) = settings.as_object_mut() {
        // Replace with the empirically confirmed key:
        obj.insert("<CORRECT_KEY>".to_string(), serde_json::json!(true));
        // Remove the dead key if present (cleanup):
        obj.remove("lspRecommendationDisabled");
    }
    std::fs::write(&settings_path, serde_json::to_string(&settings)?)
}
```

### Step 2 — Fix the no-host-settings path

Split LSP suppression from authentication so it applies even when `~/.claude.json` doesn't
exist. The current early-return in `prepare()` skips everything including LSP suppression.

Introduce a **guaranteed minimal settings injection**. Add a new function:

```rust
/// Creates a minimal `~/.claude/settings.json` with LSP recommendations disabled.
/// Called as a fallback when full `HostSettings::prepare()` returns `None`.
fn create_minimal_claude_dir(dir: &Path) -> std::io::Result<()> {
    std::fs::create_dir_all(dir)?;
    disable_lsp_recommendations(dir)
}
```

Then update the callers (`chat.rs`, `tui/mod.rs`) that currently do:

```rust
let host_settings = docker::HostSettings::prepare(&agent_name);
```

to fall back to a minimal settings-only variant when `prepare` returns `None`:

```rust
let host_settings = docker::HostSettings::prepare(&agent_name)
    .or_else(|| docker::HostSettings::prepare_minimal());
```

Add `HostSettings::prepare_minimal()` to `docker/mod.rs`:

```rust
/// Creates a minimal HostSettings with only LSP suppression — no auth forwarding.
/// Used when the host has no `~/.claude.json` (e.g. first-time setup).
pub fn prepare_minimal() -> Option<Self> {
    let temp_dir = tempfile::TempDir::new().ok()?;
    let config_path = temp_dir.path().join("claude.json");
    // Write an empty (but valid) config so the bind-mount target exists.
    std::fs::write(&config_path, "{}").ok()?;
    let claude_dir_path = temp_dir.path().join("dot-claude");
    std::fs::create_dir_all(&claude_dir_path).ok()?;
    disable_lsp_recommendations(&claude_dir_path).ok()?;
    Some(HostSettings {
        _temp_dir: Some(temp_dir),
        config_path,
        claude_dir_path,
    })
}
```

> **Note**: `prepare_minimal()` only applies when `prepare()` returns `None` (no host
> `~/.claude.json`). When `prepare()` succeeds, its flow already calls
> `disable_lsp_recommendations`, so no change is needed to that path.

### Step 3 — Also apply to `prepare_to_dir` (already present, no change needed)

`prepare_to_dir` already calls `disable_lsp_recommendations`. After the key fix in Step 1 it
will work correctly. No structural change needed.

## Test Plan

### Unit tests in `src/docker/mod.rs`

1. **`disable_lsp_recommendations_creates_file_with_correct_key`**
   - Create a temp dir with no `settings.json`.
   - Call `disable_lsp_recommendations`.
   - Assert the file exists and contains the correct key set to `true`.

2. **`disable_lsp_recommendations_preserves_existing_settings`**
   - Write a `settings.json` with unrelated keys (e.g. `{"theme": "dark"}`).
   - Call `disable_lsp_recommendations`.
   - Assert existing keys are preserved and the LSP key is added.

3. **`disable_lsp_recommendations_overwrites_false`**
   - Write a `settings.json` with the LSP key set to `false`.
   - Call `disable_lsp_recommendations`.
   - Assert it is overwritten to `true`.

4. **`prepare_minimal_returns_valid_host_settings`**
   - Call `HostSettings::prepare_minimal()`.
   - Assert it returns `Some`.
   - Assert `config_path` exists and contains valid JSON.
   - Assert `claude_dir_path` exists.
   - Assert `claude_dir_path/settings.json` contains the LSP key set to `true`.

5. **`host_settings_lsp_key_present_after_prepare`** *(dev-machine gated)*
   - Gate on `~/.claude.json` existing (same pattern as `host_settings_prepare_sanitizes_oauth`).
   - Call `HostSettings::prepare("claude")`.
   - Read `claude_dir_path/settings.json`.
   - Assert LSP key is `true`.

### Regression check

- Run `make test` — all existing tests must pass without change.

## Out of Scope

- Installing language servers in the container Dockerfile to eliminate the root trigger.
- Suppression for non-Claude agents (Codex, opencode, nanoclaw). They have no equivalent
  settings surface; no change needed.
- Modifying the `--bare` flag path — too aggressive (disables plugins and hooks).
