## 1. `draw_remote_picker` uses byte length instead of character length [trivial]

**File:** `src/tui/render.rs`, `draw_remote_picker` function (~line 1755)

```rust
let content_width = items.iter().map(|s| s.len()).max().unwrap_or(0) as u16;
```

`str::len()` returns the number of bytes, not Unicode scalar values.  For
ASCII-only content (session IDs, workdir paths) this is fine, but a path
containing multi-byte characters (rare but possible) causes the popup to be
wider than necessary.  It will never clip content — only over-allocate — so
there is no functional regression.

**Suggested fix:** Use `s.chars().count()` instead of `s.len()` to match the
intent of "fit the widest item."

## 2. Startup cleanup logs aggregate count only, not per-session detail [minor]

**File:** `src/commands/headless/server.rs`, `start_server` function (~line 1049)

```rust
Ok(n) => tracing::info!(deleted = n, "Cleaned up closed sessions older than 24h"),
```

The spec asks for a per-session log line (e.g., "deleted stale session \<id\>
and \<n\> linked command records") to make audit logs more useful.  The
current code only logs the total count.  The spec wording was "something
like", so this is aspirational.

**Suggested fix:**  
1. Extend `db::delete_closed_sessions_older_than` to return `Vec<(String, usize)>`
   (session_id, commands_deleted) instead of `usize`.
2. Log each entry before deletion:
   `tracing::info!(session_id = %sid, commands = n, "Purging stale closed session");`

## 3. `resolve_api_key` host-match normalisation is case-sensitive on path [minor]

**File:** `src/commands/remote.rs`, `resolve_api_key`

The implementation normalises hosts to lowercase before comparison
(`normalize` calls `.to_lowercase()`), which correctly handles host/scheme
case-insensitivity.  However, the *path* component of a URL is case-sensitive
by spec; since `defaultAddr` is typically `http://host:port` (no path) this
is unlikely to cause issues in practice.  Document that `defaultAddr` must
not include a path suffix if strict matching is desired.

## 4. `RemoteSaveDirConfirm` dialog handler ignores `dir` and `remote_addr` [trivial]

**File:** `src/tui/input.rs`, `handle_remote_save_dir_confirm` (~line 1621)

```rust
let _ = dir;
let _ = remote_addr;
```

Both values are suppressed with `let _` because the actual session-start
execution reads them from `PendingCommand`.  The dialog receives them as
parameters purely for symmetry with the other dialog handlers.  This is
intentional and correct, but the dead variables could be replaced by
`_dir: String, _remote_addr: String` parameters to make the intent explicit.

## 5. `fetch_and_show_session_picker` pre-selects last session but `fetch_and_show_session_kill_picker` does not [minor]

**File:** `src/tui/mod.rs`, `fetch_and_show_session_kill_picker` (~line 1763)

The session-run picker pre-selects the most-recently-used session
(`last_remote_session_id`).  The kill picker always starts at index 0.  This
is consistent with the current spec but could be improved for UX: if the user
just ran a command in session X and now wants to kill it, having X pre-selected
saves one keypress.
